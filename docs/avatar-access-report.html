✅ What we accomplished

Three layers, smallest-blast-radius first. Together they take the avatar from “read the brain / calendar / web” to “send texts, drive the whole Mac, and gain new capabilities by config alone.”

1 · iMessage + shell tools done

commit 2f88506
  • send_imessage(to, body) — sends a text via Messages.app
  • read_imessage(contact?, limit?) — reads recent threads from chat.db
  • run_shell(command) — the broad “do anything on the Mac” lever
  • Hard deny-list (rm -rf /, mkfs, dd, curl|sh, SSH keys, Keychain, sudo…)
  • confirm gate + structured audit log + timeouts + output cap

2 · MCP client done

commit ee90fe4
  • mcp.rs — JSON-RPC-over-stdio client (spawn → initialize → list/call → close)
  • Commands: mcp_list_tools, mcp_call_tool, mcp_probe
  • Config: McpServer { name, command, args, env, enabled }
  • Agent auto-discovers MCP tools & merges them into the model’s tool list
  • Settings UI to add / test / enable / remove servers

3 · UI approval modal done

commit b2dfc27
  • Real pop-up before run_shell / send_imessage run
  • Shows the exact command / recipient + message
  • Authoritative: ignores any model-set confirm=true — only your click runs it
  • Injection-safe: a malicious email can’t self-approve a command
  • Stop declines a pending request and unblocks the loop

Why this is the “in bulk” answer core goal

  • Before: each new capability = edits in 6 files (Rust + TS + daemon)
  • Now: a new MCP server entry = a whole tool set, zero code
  • Filesystem, iMessage, Notes, GitHub, Slack… add by config
  • run_shell covers the long tail nothing else does

The new confirm flow (interactive turns)

model wants to run a command │ ▼ deny-list? ──yes──▶ hard block (cannot be overridden) │ no ▼ UI MODAL ──▶ you click “Approve & run” ──▶ runs, returns output │ └──▶ you click “Decline” / Stop ──▶ not run, model told you declined
